store the min (3) in a root, then continue with 5, 8, 10, 20, 50, 80

Goal: Optimize the lookup operation, not the insert time.



This will keep the tree balanced on L and R, making the lookup operation optimized.

Need to calculate the equilibrium (balance) factor - BF for each node



Heigh(Left/Right Subtree):

GE = H(LS) - H(RS) (or viceversa)

GE belongs to {-1, 0, 1}



Height computation:

if H(AVL\* root) return (root == NULL) ? 0 : 1 + max(H(root->left), H(root->right));

O(log(n)) will be kept for the lookup function as well as the insertion and deletion operations

since O(log(n)) + c = O(log(n))



!Only insertion and deletion can cause the tree to become unbalanced!

To counteract this, we need to apply rotations.

A full binary tree is the only tree that is perfectly balanced (as in all GE are 0 on all levels)



\*If the root has GE = -1/0/1, it is considered balanced, but not perfectly balanced.

\*when you work with binary trees, we should never leave them unbalanced.

\*to re-balance, insert/delete in a specific way





when doing the height computation, you take the number of nodes (if any) on the right side and substract the number of nodes on the left side.
